SCRIPTS NODE.JS - ESTANDAR DE ROBUSTEZ:
Cuando sea necesario crear scripts temporales (.cjs), aplicar SIEMPRE este template:

```javascript
const fs = require('fs');
const path = require('path');

// === CONFIGURACION ===
const TARGET_FILE = path.join(__dirname, 'Proyecto', 'archivo.html');

// === VALIDACION PREVIA ===
if (!fs.existsSync(TARGET_FILE)) {
    console.error(`✗ Error: No se encuentra el archivo ${TARGET_FILE}`);
    process.exit(1);
}

// === LECTURA SEGURA ===
let content;
try {
    content = fs.readFileSync(TARGET_FILE, 'utf8');
} catch (err) {
    console.error(`✗ Error al leer archivo: ${err.message}`);
    process.exit(1);
}

// === BACKUP AUTOMATICO ===
const backupFile = TARGET_FILE + '.backup';
try {
    fs.writeFileSync(backupFile, content, 'utf8');
    console.log(`✓ Backup creado: ${backupFile}`);
} catch (err) {
    console.error(`✗ Error al crear backup: ${err.message}`);
    process.exit(1);
}

// === TRANSFORMACIONES ===
const originalLength = content.length;
let modified = false;

// Ejemplo de reemplazo con validacion
const oldPattern = 'patron-a-buscar';
if (content.includes(oldPattern)) {
    content = content.replace(oldPattern, 'nuevo-patron');
    modified = true;
    console.log(`✓ Patron reemplazado`);
} else {
    console.warn(`⚠ Patron no encontrado: "${oldPattern}"`);
}

// === VALIDACION POST-TRANSFORMACION ===
if (!modified) {
    console.warn('⚠ No se realizaron cambios. Archivo sin modificar.');
    process.exit(0);
}

if (content.length === 0) {
    console.error('✗ Error critico: El contenido resultante esta vacio. Abortando.');
    process.exit(1);
}

// === ESCRITURA SEGURA ===
try {
    fs.writeFileSync(TARGET_FILE, content, 'utf8');
    console.log(`✓ Archivo actualizado: ${TARGET_FILE}`);
    console.log(`  Tamano original: ${originalLength} bytes`);
    console.log(`  Tamano nuevo: ${content.length} bytes`);
} catch (err) {
    console.error(`✗ Error al escribir archivo: ${err.message}`);
    console.log(`  Restaurando desde backup...`);
    try {
        const backup = fs.readFileSync(backupFile, 'utf8');
        fs.writeFileSync(TARGET_FILE, backup, 'utf8');
        console.log(`✓ Archivo restaurado desde backup`);
    } catch (restoreErr) {
        console.error(`✗ Error critico al restaurar: ${restoreErr.message}`);
    }
    process.exit(1);
}

// === LIMPIEZA OPCIONAL ===
// Descomentar para eliminar backup automaticamente tras exito
// fs.unlinkSync(backupFile);

console.log('✓ Operacion completada exitosamente');
```

REGLAS OBLIGATORIAS PARA SCRIPTS:
1. SIEMPRE crear backup antes de modificar (.backup).
2. SIEMPRE validar que el archivo existe antes de leer.
3. SIEMPRE validar que el patron existe antes de reemplazar.
4. SIEMPRE validar que el contenido resultante no esta vacio.
5. SIEMPRE usar try-catch para operaciones de I/O.
6. SIEMPRE reportar tamanos de archivo (antes/despues).
7. SIEMPRE usar console.log con simbolos (✓ ✗ ⚠) para claridad.
8. NUNCA usar `content.replace()` sin verificar que el patron existe primero.
9. NUNCA escribir archivo si el contenido resultante es sospechoso (vacio, muy pequeno).
10. Implementar restauracion automatica desde backup si la escritura falla.

MANEJO DE ERRORES COMUNES:
- "oldStr not found": Leer el archivo primero con read tool, copiar texto EXACTO (espacios/tabs).
- Script vacia el archivo: Falta validacion de contenido antes de writeFileSync.
- Script no encuentra patron: Agregar console.log del contenido antes del replace para debug.
- Multiples reemplazos fallan: Dividir en operaciones atomicas, una por script.

LIMPIEZA POST-SCRIPT:
- Eliminar scripts temporales (.cjs) inmediatamente tras ejecucion exitosa.
- Eliminar backups (.backup) solo si el usuario confirma que todo funciona.
- Comando recomendado: `Remove-Item script.cjs` (PowerShell) o `rm script.cjs` (bash).

--- ARQUITECTURA OWS STORE (TAURI v2) ---

OWS Store fue migrado de Electron a Tauri v2 + Rust. Es el UNICO proyecto en esta arquitectura;
todos los demas proyectos del ecosistema (Ocean Cinemas, Wild Destiny, Ocean Pay, etc.) siguen en Electron o web.

ESTRUCTURA DEL PROYECTO:
- `index.html` + `dist/index.html`: Frontend HTML/JS/CSS con Tauri shim.
- `src-tauri/src/lib.rs`: Backend Rust (~720 lineas) con todos los comandos IPC.
- `src-tauri/tauri.conf.json`: Configuracion Tauri (ventana, updater, NSIS, icons).
- `src-tauri/Cargo.toml`: Dependencias Rust (tauri v2, tauri-plugin-updater, reqwest, etc.).
- `src-tauri/updater-keys.json`: Clave privada para firmar updates (SECRETO).
- `src-tauri/updater-keys.json.pub`: Clave publica para verify updates.

Tauri SHIM (en index.html, linea 10):
- Reemplaza al preload.js de Electron.
- Detecta automaticamente Tauri v1 (`window.__TAURI__`) o v2 (`window.__TAURI_INTERNALS__`).
- Expone `window.owsUpdater` con metodos IPC identicos a Electron para compatibilidad del frontend.
- Propiedades: `isElectron: true` (compat), `isTauri: true` (deteccion).

MECANISMO DE UPDATE (SELF-UPDATE):
- NO usa electron-updater ni latest.yml.
- Frontend consulta GitHub Releases API directamente para detectar version nueva.
- Descarga e instalacion via comando Rust `install_external_installer`.
- Manifest de update: `ows-store-tauri-update.json` en raiz de owsdatabase.
- Firma del installer: `cargo tauri signer sign`.

BUILD:
- `cargo tauri build` produce: `ows-store.exe` y `OWS Store_<cargo-version>_x64-setup.exe`.
- NSIS installer con tema oscuro via post-build `scripts/patch-nsis-installer.ps1`.
- Version semver en Cargo.toml (0.1.0); version real YYYY.M.D-tHHMM en APP_VERSION del frontend y tag GitHub.

COMANDOS RUST DISPONIBLES (via invoke IPC):
- `check_for_updates`: Busca nueva version en GitHub Releases.
- `install_update`: Descarga e instalacion via updater plugin (con fallback manual).
- `install_external_installer`: Descarga un instalador externo y lo ejecuta (usado para self-update).
- `get_app_version`: Devuelve la version compilada.
- `get_wns_channel_uri`: Obtiene URI de canal WNS (notificaciones).
- `show_system_notification`: Muestra notificacion del sistema.
- `open_external_url`: Abre URL en browser externo.
- `resolve_installed_apps_batch`: Verifica apps instaladas en el sistema.
- `launch_installed_app`: Lanza una app instalada.
- `uninstall_installed_app`: Desinstala una app.

TRAY ICON:

--- ADVERTENCIA CRITICA: FTFY Y MOJIBAKE ---

NUNCA usar `ftfy` (Python) para corregir mojibake en archivos HTML con JavaScript embebido.
ftfy convierte comillas normales (", ') en comillas tipograficas (", ", ', ') que rompen sintaxis JavaScript.

SINTOMAS:
- SyntaxError: Unexpected string
- Decenas de miles de comillas tipograficas introducidas
- Codigo JavaScript invalido

SOLUCION CORRECTA PARA MOJIBAKE:
1) Para texto mojibake (Ã¡ -> á, etc):
   - Usar `mojibak` (npm): `import { recover } from 'mojibak'`
   - Aplicar solo a strings especificos, NO a todo el archivo

2) Para emojis mojibake (ðŸ" -> 📦):
   - Reemplazo manual directo con `str.replace()` o `strReplace` en write tool
   - NO usar ftfy ni conversiones de encoding automaticas

3) Si ftfy ya se ejecuto:
   - Restaurar archivo: `git checkout <archivo>`
   - Aplicar correcciones quirurgicas con mojibak/replace manual
   - Verificar sintaxis JavaScript antes de commit

--- CHANGELOG VIA API EN DOS IDIOMAS ---

Cuando el usuario diga "Changelog via API en ambos idiomas" (o similar), publicar DOS entradas separadas en la timeline:
una en ingles y otra en espanol, usando el mismo endpoint POST /ows-news/updates con x-ows-admin-token.

REGLA: usar `banner_meta.sync_key` distinto para cada idioma para que sean entradas independientes.
Convencion de sync_key: `<slug>-changelog-<version-o-fecha>-en` y `<slug>-changelog-<version-o-fecha>-es`.

ESTRUCTURA DE CADA ENTRADA:

Entrada EN (ingles):
  POST https://owsdatabase.onrender.com/ows-news/updates
  Header: x-ows-admin-token: <OWS_ADMIN_SECRET>
  Body:
  {
    "title": "<Titulo en ingles>",
    "description": "<Descripcion breve en ingles>",
    "changes": ["Cambio 1 en ingles", "Cambio 2 en ingles", ...],
    "project_slugs": ["<slug>"],
    "project_names": ["<Nombre visible>"],
    "entry_type": "changelog",
    "platforms": ["windows"] | ["windows", "android"] | ["android"],
    "is_active": true,
    "priority": 10,
    "banner_meta": { "sync_key": "<slug>-changelog-<version>-en" }
  }

Entrada ES (espanol):
  POST https://owsdatabase.onrender.com/ows-news/updates
  Header: x-ows-admin-token: <OWS_ADMIN_SECRET>
  Body:
  {
    "title": "<Titulo en espanol>",
    "description": "<Descripcion breve en espanol>",
    "changes": ["Cambio 1 en espanol", "Cambio 2 en espanol", ...],
    "project_slugs": ["<slug>"],
    "project_names": ["<Nombre visible>"],
    "entry_type": "changelog",
    "platforms": ["windows"] | ["windows", "android"] | ["android"],
    "is_active": true,
    "priority": 10,
    "banner_meta": { "sync_key": "<slug>-changelog-<version>-es" }
  }

RESULTADO EN OWS STORE:
- Aparecen dos entradas en la ficha del proyecto: una en ingles y otra en espanol.
- El toggle EN/ES en OWS Store permite al usuario elegir cual ver.
- Ambas entradas son independientes en la DB (sync_key distinto).
- Si se necesita actualizar una sola, re-POST con el mismo sync_key la sobreescribe (upsert).

EJEMPLO REAL (Floret Shop, version 2026.4.28-t2358):
  sync_key EN: "floretshop-changelog-2026.4.28-t2358-en"
  sync_key ES: "floretshop-changelog-2026.4.28-t2358-es"

NOTA: el campo `description_es` / `changes_es` en el body es un mecanismo alternativo
que guarda ambos idiomas en UNA sola entrada (en el campo details JSONB).
Ese mecanismo tambien funciona pero requiere que el servidor este actualizado.
El metodo de DOS ENTRADAS SEPARADAS es mas robusto y no depende de la version del servidor.


--- API DE NOTICIAS OWS STORE (PARA AGENTES) ---

ENDPOINT PRINCIPAL: POST /ows-store/news
URL: https://owsdatabase.onrender.com/ows-store/news
Header requerido: x-ows-admin-token: <OWS_ADMIN_SECRET>

PROPOSITO:
Crear noticias/anuncios en la timeline central de OWS Store (tabla: ows_store_timeline, kind='news').
Estas noticias aparecen en la seccion de noticias de OWS Store y pueden asociarse a proyectos especificos.

PARAMETROS DEL BODY (JSON):
{
  "seed_key": "string",              // Identificador unico (ej: "ocean-cinemas-update-2026-05-04")
  "kind": "news",                    // Tipo de entrada: "news", "event", "changelog", "banner"
  "project_slug": "string",          // Slug del proyecto relacionado (opcional, puede ser null para noticias globales)
  "title": "string",                 // Titulo de la noticia (requerido)
  "description": "string",           // Descripcion/contenido principal (requerido)
  "content_lines": ["string"],       // Array de lineas adicionales de contenido (opcional, puede ser [])
  "platforms": ["windows", "android"], // Plataformas afectadas (opcional)
  "is_active": true,                 // Si la noticia esta activa/visible (default: true)
  "priority": 0                      // Prioridad de ordenamiento (opcional, default: 0)
}

CAMPOS OPCIONALES ADICIONALES:
- "banner_meta": {}                  // Metadata adicional para banners/eventos (JSON)
- "published_at": "ISO_DATE"         // Fecha de publicacion (default: now)

EJEMPLO DE USO (Node.js):
```javascript
const https = require('https');

const data = JSON.stringify({
  "seed_key": "ocean-cinemas-recommendations-fix-2026-05-04",
  "kind": "news",
  "project_slug": "ocean-cinemas",
  "title": "Ocean Cinemas: Recommendations System Improved",
  "description": "Fixed recommendations to show diverse series instead of repeating episodes from the same series. Reading times now accurate based on actual content length.",
  "content_lines": [],
  "platforms": ["windows"],
  "is_active": true
});

const options = {
  hostname: 'owsdatabase.onrender.com',
  port: 443,
  path: '/ows-store/news',
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'x-ows-admin-token': process.env.OWS_ADMIN_SECRET,
    'Content-Length': data.length
  }
};

const req = https.request(options, (res) => {
  let body = '';
  res.on('data', (chunk) => body += chunk);
  res.on('end', () => {
    console.log('Status:', res.statusCode);
    console.log('Response:', body);
  });
});

req.on('error', (error) => console.error('Error:', error));
req.write(data);
req.end();
```

RESPUESTA EXITOSA (200):
{
  "success": true,
  "news": {
    "id": 581,
    "seed_key": "ocean-cinemas-recommendations-fix-2026-05-04",
    "title": "Ocean Cinemas: Recommendations System Improved",
    "description": "...",
    "is_active": true,
    "published_at": "2026-05-04T23:45:23.378Z",
    ...
  }
}

ERRORES COMUNES:
- 401 Unauthorized: Token admin ausente o incorrecto
  - Verificar que x-ows-admin-token este en headers
  - Verificar que el valor sea correcto (OWS_ADMIN_SECRET)
- 400 Bad Request: Parametros faltantes o invalidos
  - Verificar que title y description esten presentes
  - Verificar formato JSON valido

UBICACION DEL TOKEN ADMIN:
- Archivo local: C:\Users\hachi\OneDrive\Escritorio\Ocean and Wild Studios\secrets\workspace-secrets
- Formato: STUDIO_SECRET: <valor>
- Variable de entorno: $env:OWS_ADMIN_SECRET (PowerShell)
- NUNCA commitear este archivo ni el token

OTROS ENDPOINTS RELACIONADOS:
- GET /ows-store/news - Listar todas las noticias
- GET /ows-store/news/:id - Obtener noticia especifica
- PATCH /ows-store/news/:id - Actualizar noticia existente
- DELETE /ows-store/news/:id - Eliminar noticia (requiere admin token)

INTEGRACION CON CHANGELOG:
Cuando se publica un changelog via API (POST /ows-store/projects/:slug/version),
se puede crear una noticia asociada para anunciar la actualizacion:
1. Publicar changelog con PATCH /ows-store/projects/:slug/version
2. Crear noticia con POST /ows-store/news referenciando el project_slug
3. La noticia aparecera en la timeline de OWS Store asociada al proyecto

REGLAS OPERATIVAS:
- Usar este endpoint cuando el usuario pida "crear noticia", "publicar anuncio", o "agregar a OWS Store news"
- NO usar para changelogs automaticos de releases (esos van por PATCH /ows-store/projects/:slug/version)
- Ideal para anuncios manuales, actualizaciones de desarrollo, comunicados especiales


--- SISTEMA DE RECUPERACION Y BACKUP A GITHUB (OWS RECOVER) ---

Para prevenir pérdida de datos ante fallos del sistema, se ha configurado un repositorio de respaldo unificado.

- Rama principal: main

REPOSITORIOS LOCALES:
- Proyectos Web & Secrets: C:\Users\hachi\OneDrive\Escritorio\owsrecover
- Proyectos Unity: C:\Users\hachi\OneDrive\Escritorio\owsrecover-unity

SCRIPTS DE RESPALDO:
1. Script PowerShell UNICO: `C:\Users\hachi\OneDrive\Escritorio\Ocean and Wild Studios\scripts\backup-to-github.ps1`
   Sincroniza todos los proyectos Web activos (que contienen index.html), la carpeta `secrets/`, la carpeta `OWS Store/scripts`, y los 4 proyectos de Unity (Animaciones, Bomberman, Stupid Zombies, Tower Defense) excluyendo archivos basura pesados (node_modules, Library, dist, APKS).
   Ademas genera/actualiza `BACKUP_STATUS.json` en la raiz de owsrecover, que es la fuente autoritativa que el Centro de Control OWS usa para verificar los respaldos.
2. Lanzador Windows: `C:\Users\hachi\OneDrive\Escritorio\Ocean and Wild Studios\OWS Store\scripts\ejecutar-backup.bat`

--- REGISTRAR VERSION EN CADA BACKUP (WIP vs RELEASE) ---

El script acepta un parametro `-version` (string, opcional):
- Si se pasa `-version X`            => se registra X como la version real del release.
- Si NO se pasa `-version`           => se registra `WIP-YYYY.M.D-tHHMM` (timestamp Uruguay) como marcador de cambios de preparacion.

En CUALQUIER caso (release o cambios de preparacion), se debe correr el script. Esto es para que el Centro de Control OWS pueda separar visualmente:
- `🛠 WIP: WIP-2026.6.7-t1230`  (cambios hechos, todavia no son release)
- `🚀 Release: 2026.6.7-t1230`  (cambio asociado a un release publicado)

REGLAS OPERATIVAS PARA EL AGENTE:
- Si el usuario pidio CAMBIOS pero NO pidio publicar una nueva version, se debe correr:
    ejecutar-backup.bat
  (sin parametro -version, asi queda registrado como WIP con la fecha/hora Uruguay).
- Si el usuario pidio BUILD/RELEASE (y por ende hay una version explicita), se debe correr:
    ejecutar-backup.bat -version <version-real>
  (donde <version-real> es la misma version que se va a bumpear en OWS Store).
- NUNCA saltearse este paso. Es la unica manera de tener trazabilidad real
  de cuando se hicieron cambios, separados de cuando se publico un release.
- El script siempre se ejecuta, aunque no haya cambios de archivos. Asi
  el timestamp Uruguay del WIP refleja cuando se hizo la ultima sesion
  de trabajo, no solo cuando hubo cambios de codigo.

TIMESTAMP URUGUAY: el script usa la zona horaria `America/Montevideo`
para generar el WIP y para el campo `last_full_backup` del JSON. Si
la PC no tiene esa zona configurada, cae a la hora local como fallback
y avisa por consola.

DIFERENCIA CLAVE (a no confundir):
- WIP: trabajo en curso, no publicado, todavia puede cambiar. Es un
  marcador de que "se hicieron cambios el X a las Y".
- Release: cambio publicado, con version fija, ya visible para usuarios.
  Se registra cuando el usuario SI pidio publicar/bump.
